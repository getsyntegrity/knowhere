# Knowhere — Document-Native Memory Layer for AI Agents

## 产品分析 & 竞品研究报告

> 日期: 2026-03-18  
> 版本: v1.0 — 合并 AI Input 评估 + Agent Memory 深度分析

---

## 1. 产品定位

### 1.1 一句话定义

> **Knowhere = 基于深度文档解析的 Agent 记忆基础设施**
>
> 不同于从对话中构建记忆（Mem0），Knowhere 从真实文档中构建结构化、持久化的 Agent 记忆。对 Agent 来说，它是零配置的知识 API；对人来说，它是一个理解你所有文档的写作伙伴。

### 1.2 核心架构（双端口模型）

```
                    ┌──────────────────────────────────────────┐
                    │          Knowhere Memory Engine           │
                    │                                          │
    本地文件 ──────→│  无感扫描 → 结构化解析 → KG + 向量化      │
    (DOCX/PDF/      │  表格 · 标题层级 · 图片 · TOC · 跨文件关系 │
     XLSX/PPTX)     │                                          │
                    └──────────────┬───────────────┬───────────┘
                                   │               │
                      ┌────────────┴───┐   ┌───────┴──────────┐
                      │   Agent 端口    │   │   Human 端口      │
                      │   (REST API)    │   │   (选中→Hotkey→   │
                      │   零 UI         │   │    浮窗/Typing)   │
                      │   纯记忆服务    │   │   写作辅助出口     │
                      └────────────────┘   └──────────────────┘
```

**核心洞察**：同一套记忆引擎服务两类消费者。Agent 需要 `POST /retrieve` + structured data；人需要一个 UI 出口来"看到"和"使用"记忆。这种双端口模型使得 B2D（服务开发者）和 B2C（服务终端用户）可以独立验证、互相导流。

## 2. 技术方案拆解

### 2.1 输入端 — 无感知识库（已完成）

Knowhere API 的解析管线已经实现：

| 能力 | 实现 |
|------|------|
| **自动扫描** | 监控 ~/Documents、~/Desktop、~/Downloads 等目录 |
| **多格式解析** | DOCX (标题/层级/表格/图片/TOC)、PDF (精准模式+快速路径)、XLSX (子表合并/MultiIndex)、PPTX (iLoveAPI 转换)、Markdown/TXT/图片 |
| **结构化提取** | 表格理解、标题层级优化 (BFS layout_parser)、嵌入图片提取+LLM 摘要、TOC 利用 |
| **向量化** | Embedding → 向量存储，支持语义检索 |
| **知识图谱** | ConnectTo 跨文件关系构建（长度加权评分、字符去重、倒排索引），chunk→file 聚合 |
| **智能路由** | Agentic Profiler 识别 PDF 类型（扫描件/电子版/PPT转换），自动选择解析路径 |

### 2.2 输出端 — Human 端口（待实现）

AI Input Layer 定义的交互流程：

```
用户选中文本 → Hotkey (Option+Enter) → 捕获选中内容
                                           ↓
                                    RAG 检索知识上下文
                                           ↓
                                    LLM 生成下一段
                                           ↓
                              ┌────────────┼────────────┐
                              │            │            │
                         浮窗预览      Typing 注入    剪贴板
                         (MVP)        (锦上添花)     (fallback)
```

### 2.3 输出端 — Agent 端口（核心价值）

```
Agent App ──POST /retrieve──→ Knowhere API ──→ structured chunks + KG context
                                                   ↓
                                           Agent 直接消费记忆
                                           (无 UI、无 Typing)
```

## 3. 行业盲区：文档级记忆 vs 对话级记忆

### 3.1 为什么这是一个盲区

当前 Agent Memory 赛道（Mem0/Zep/LangMem）的共同假设是：**记忆来自对话**。但现实中：

> 一个工程师 90% 的知识不在他说过的话里，而在他桌面上的 PDF/DOCX/XLSX 里。

| | 对话级记忆 (Mem0/Zep) | 文档级记忆 (Knowhere) |
|---|------|---|
| **记忆来源** | 从 chat 中提取 → 记忆是"说出来的" | 从文档中解析 → 记忆是"写下来的" |
| **解析对象** | 自然语言句子 → atomic facts | 复杂文档 → 结构化数据 (表格/层级/图片) |
| **知识密度** | 低 — 一次对话提取几个 fact | 高 — 一个 DOCX 产出几百个结构化 chunk |
| **冷启动** | 需要多轮对话"教会" Agent | 文件扫描即获得完整知识 |
| **消费方式** | 仅 Agent API | Agent API + Human 写作接口 |

### 3.2 这意味着什么

- Agent 不需要先"聊过"才能拥有知识 — 文件扫描即就绪
- 知识密度高几个数量级 — 50 页合同远比 10 轮对话信息丰富
- 结构化程度天壤之别 — 表格/层级/跨文件关系 vs flat text
- **市场上没有人** 把"深度文档解析 + KG + Agent Memory API"打通成一个产品

## 7. Typing 输出的定位评估

### 7.1 Typing 在架构中的位置

Typing 是 Human 端口的 **一个输出形态**，不是核心架构组件：

```
                    Human 端口
                        │
           ┌────────────┼────────────┐
           │            │            │
      浮窗预览      Typing 注入    API 直接返回
      (MVP 优先)    (锦上添花)     (开发者用)
```

### 7.2 三种输出方式对比

| 方案 | 优势 | 劣势 | 优先级 |
|------|------|------|--------|
| **浮窗预览** | 简单可靠、即时、零权限 | 需手动粘贴 | ⭐ MVP |
| **Typing 注入** | 视觉炫酷、跨应用、演示效果强 | macOS Accessibility 权限、跨应用兼容性风险、延迟放大 | Phase 2 |
| **编辑器插件** (Cursor 模式) | 最自然、零摩擦 | 需编辑器级集成，无法跨应用 | Phase 3 |

### 7.3 Typing 注入的具体风险

| 风险 | 详情 |
|------|------|
| **OS 权限** | macOS Sonoma+ 对 Accessibility 权限审核越来越严，授权流程不流畅会杀死转化率 |
| **跨应用兼容** | 模拟键入在 Chrome/Notion/Slack/Word 表现不一，部分 Electron 应用会吞掉模拟事件 |
| **延迟感知** | RAG (200ms) + LLM (1-3s) + Typing 动画 = 实际体感 3-5s，用户等待焦虑 |
| **交互范式** | 2025-2026 市场已往"主动预测"(Cursor Tab 模式) 走，被动触发有摩擦差距 |

### 7.4 结论

Typing 作为锦上添花的定位 **完全正确**：
- 花 20% 工程量获得 80% 产品感知提升
- 演示效果强（投资人/用户面前"AI 在为你打字"）
- 但不应是 MVP 的默认方式，也不是核心架构的一部分

## 10. 最终结论

### 核心判断

1. **你手里握着一个市场盲区的解法** — 文档级 Agent 记忆。Mem0/Zep/LangMem 全部假设"记忆来自对话"，你从文档构建记忆，知识密度和结构化程度碾压。

2. **双端口模型是真正的产品智慧** — 同一引擎同时服务 Agent (API) 和 Human (写作)，B2D 和 B2C 独立验证、交叉导流。

3. **Typing 作为锦上添花完全正确** — 核心价值在记忆引擎，不在打字动画。MVP 用浮窗预览验证价值，Typing 作为后续炫酷特性。

4. **核心精力应聚焦在**：

```
┌─────────────────────────────────────────────────────┐
│  Memory Engine 质量      >>>  Human 端口 UI 花哨度    │
│  KG 洞察 + 检索精度      >>>  Typing 注入效果         │
│  Agent 生态集成 (SDK/MCP) >>>  跨应用兼容性            │
└─────────────────────────────────────────────────────┘
```

### 一句话

> **Knowhere 的护城河不是 Typing，而是那条从复杂文档到结构化 Agent 记忆的解析管线。这是市场上没有人做到的事。**
